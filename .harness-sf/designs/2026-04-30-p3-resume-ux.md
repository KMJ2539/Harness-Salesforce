# P3 — resume UX (세션 손실 후 재진입)

## Why

dispatch 가 중간에 끊기는 시나리오는 흔하다 — 세션 종료, Ctrl+C, sub-skill 실패 후 사용자가 자리를 비움.

상태는 이미 dispatch-state JSON 에 모두 보존되어 있다. 부족한 건 **UX 진입점**: 사용자가 다시 `/sf-feature` 를 호출했을 때 자동으로 미완 작업을 인지하고 이어 받을 경로가 명시되어 있지 않다. 현재는 처음부터 다시 시작하거나, 사용자가 수동으로 dispatch-state-cli 를 호출해야 한다.

P0 (canonical state 단일화) + P1 (phase / current 가시화) 가 끝나면 resume 의 source signal 이 정확해지므로, 그 위에 UX 만 얹으면 된다.

## What

세 가지 진입점:

1. **자동 감지**: `/sf-feature` 실행 시 미완 dispatch-state 가 있으면 즉시 사용자에게 `[Resume <slug> / Start new / Show status]` 제시.
2. **명시적 명령**: `/sf-feature resume <slug>` — Step 1~5 (intent / decomposition / design / review / approval) 스킵하고 Step 6 (dispatch) 부터 재개.
3. **idempotent 재실행**: artifact 단위 재실행이 안전 — 이미 done 인 artifact 는 skip, in_progress 는 이어서 (또는 사용자에게 재시도 / 건너뛰기 / 중단 선택).

## How (변경 파일)

| 파일 | 변경 |
|---|---|
| `templates/skills/sf-feature/SKILL.md` | 신규 `Step 0.3: resume detection` 추가. dispatch-state 디렉터리 스캔 → 미완 (`pending`/`in_progress`/`failed` artifact ≥ 1) feature 가 있으면 사용자에게 옵션 제시. |
| `templates/skills/sf-feature/SKILL.md` | resume 분기 흐름: Step 0.3 → Step 5.2 (approval sentinel 만료 여부 확인, 만료 시 재발급) → Step 6 (dispatch) → Step 7.5 (validate). Step 1~5 는 명시적으로 skip. |
| `templates/skills/sf-feature/SKILL.md` | 사용 예시 섹션에 `/sf-feature resume order-module` 추가. |
| `templates/hooks/_lib/dispatch-state-cli.js` | `resume <slug>` 서브커맨드 추가: failed artifact 를 `pending` 으로 되돌리거나 (사용자 선택), 다음 미완 artifact id 를 stdout 으로 반환. |
| `templates/hooks/_lib/dispatch-state-cli.js` | `list-incomplete` 서브커맨드 추가: 미완 feature slug 목록 반환 (resume detection 이 사용). |
| 5개 artifact 스킬 (`sf-apex/sf-lwc/sf-aura/sf-sobject/sf-field`) `SKILL.md` | delegated mode 진입 시 idempotency 체크: 동일 artifact id 가 이미 done 이면 즉시 종료, in_progress 면 사용자에게 [Continue / Restart / Skip] 제시. |

### resume detection 시그널 (Step 0.3)

```
candidates = dispatch-state-cli list-incomplete
  → returns slugs where ANY artifact.status ∈ {pending, in_progress, failed}
  → 단, 마지막 갱신이 7일 초과면 stale 로 간주, 자동 제시 안 함 (사용자가 명시적으로 resume 시에만)
```

stale 임계값은 conservative 하게 7일. 실제 사용 패턴 보고 조정.

### approval sentinel 재발급 정책

- design approval sentinel 만료: 본문 hash 가 그대로면 자동 재발급 (사람의 재승인 불필요 — 같은 design 에 같은 결정).
- design.md body hash 가 변경됐으면 재승인 강제 (resume 거부, 새 review/approval 사이클 필요).

## Tests

- `templates/hooks/_lib/__tests__/dispatch-state-cli.test.js` 확장:
  - `resume <slug>` 가 failed → pending 전환.
  - `list-incomplete` 가 정확한 slug 목록 반환.
  - stale (mtime > 7d) 항목은 list-incomplete 에서 제외.
- 통합 테스트: dispatch 중간에 인위적으로 state 를 `failed` 로 바꾼 후 resume 시 해당 artifact 만 재실행.
- idempotency 테스트: 동일 artifact 를 done 상태로 두고 sub-skill 재호출 → no-op.

## Done when

- 세션 손실 후 `/sf-feature` 만 호출해도 미완 작업이 자동 감지되어 옵션 제시됨.
- `/sf-feature resume <slug>` 가 design approval sentinel 만료 외에는 사용자 추가 입력 없이 dispatch 재개.
- 5개 artifact 스킬이 모두 idempotent — 같은 artifact id 로 두 번 호출해도 안전.

## Risks

- **idempotency 누락**: artifact 스킬 중 하나라도 부수효과를 두 번 일으키면 (예: SObject 이름 충돌, Apex 클래스 덮어쓰기) resume 이 위험. → 5개 스킬의 Step 2 (CREATE-vs-MODIFY 결정) 가 이미 detect-first 이므로 보호되지만, delegated mode 진입 직후 dispatch-state 의 done 체크를 추가로 강제.
- **stale state 오용**: 1주일 이상 묵은 state 를 자동 제시하면 사용자가 잊은 작업을 잘못 이어갈 수 있음. → 7일 임계값으로 자동 제시 차단, 명시적 `resume <slug>` 로만 접근 허용.
- **approval sentinel hash drift**: body hash 비교 로직이 P0 의 design.md 변경 (status 태그 제거) 과 충돌할 수 있음. → P0 가 끝난 후 새 baseline hash 로 sentinel 재발급. P3 는 P1 이후 진행이므로 자연 해결.

## Out of scope

- partial artifact resume (한 artifact 안에서 step 단위 재개). 현 단위는 artifact.
- 자동 재시도 (실패 → 동일 입력으로 즉시 재실행). 실패는 항상 사용자 결정 게이트.
- 멀티 feature 동시 dispatch (현 모델은 single active feature).
