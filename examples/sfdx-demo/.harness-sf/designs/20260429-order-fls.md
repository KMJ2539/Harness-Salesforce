## Why (Business)

Orders over $1,000 are flagged as `High Value` so the ops team can review them
inside an hour. The current handler updates `Status__c` directly without
checking whether the running user has Edit access on the field — leaking edits
in contexts where FLS should have hidden them.

## What (Scope)

- Add FLS check on `Order__c.Status__c` before update in `OrderHandler.afterInsert`.
- No changes to the eligibility threshold ($1,000) or status taxonomy.
- Out of scope: history tracking, notifications, retro-tag of past Orders.

## How (Operations)

- Use `Schema.sObjectType.Order__c.fields.Status__c.isUpdateable()` guard.
- If FLS check fails, skip update and add an `addError` on the record so the
  insert is rejected (rather than silently dropping the tag).
- Bulkified: a single guard outside the loop, since FLS is per-user not per-row.

## Edge Cases

- Bulk insert with mixed FLS visibility — guard once, applies to whole batch.
- User with Read-only on Status__c → reject insert with addError.
- Amount__c null → no status change (existing behavior preserved).

## Decisions

- Apex handler over Before-Save Flow — Flow can't easily express FLS guards
  with custom error messages.
- No new fields, no new permission sets needed.

## Reviews

# Eng Review:

- [M1] addError will roll back the entire bulk insert if any one record's user
  lacks FLS — confirm this is desired vs per-record skip.

# Security Review:

- [H1] current code updates Status__c without an FLS check — proposed fix
  closes that gap. Verify the guard is `.isUpdateable()` not just
  `.isAccessible()` (read).

# QA Review:

- [M1] need a test where the running user explicitly lacks Edit on Status__c
  (`System.runAs` with a stripped Permission Set).

## Review Resolution

- H1: confirmed using Schema.sObjectType.Order__c.fields.Status__c.isUpdateable() — Updateable, not Accessible
- M1: bulk-fail is desired (partial success is harder to reconcile downstream); test class uses System.runAs with a no-FLS PS to assert the guard fires
